в—Ҹ Here is the full flow, top to bottom.

  ---
  The Full Code Flow

  Stage 0 вҖ” Server Startup (main.py)

  When you run the server, lifespan() runs before any request is accepted:

  1. Creates directories: data/uploads/, data/chroma/, prompts/
  2. Creates SQLite tables (Document, IngestionEvent) via SQLModel
  3. OpenAI check вҖ” calls oai.models.list(), verifies both gpt-4o-mini and
  text-embedding-3-small are in your account. Hard-stops if missing.
  4. ChromaDB check вҖ” chroma.heartbeat() + creates/gets the rag_stage1
  collection with cosine distance
  5. Server is now ready

  ---
  Stage 1 вҖ” Document Upload (api/documents.py)

  Client вҶ’ POST /api/documents/upload (multipart file)

  1. Validates the file: extension must be .pdf/.epub/.docx/.md/.txt, size вүӨ 50
  MB, PDFs вүӨ 500 pages
  2. Dedup вҖ” SHA-256 hashes the content, checks if the same hash exists in
  SQLite. If yes: deletes the old Chroma chunks and SQLite rows first (clean
  re-ingest)
  3. Saves the file to data/uploads/<uuid>.<ext>
  4. Creates a Document row in SQLite with status=pending
  5. Writes two IngestionEvent rows (upload running, upload complete) so the SSE
   stream can replay them
  6. Returns 202 immediately: {document_id, filename, status: "pending"}
  7. Enqueues run_ingestion() as a FastAPI BackgroundTask вҖ” client gets the 202
  before ingestion begins

  ---
  Stage 2 вҖ” Ingestion Pipeline (ingestion/pipeline.py)

  Runs in the background after the upload response. Four steps, each writes
  IngestionEvent rows (running вҶ’ complete or failed) that the SSE endpoint
  polls:

  Step 1: Parse (ingestion/parse.py)

  - SimpleDirectoryReader([file_path]).load_data() вҖ” LlamaIndex reads the file
  - Handles PDF (page-by-page), EPUB, DOCX, Markdown, plain text
  - Returns a list of LlamaIndex Document objects, each with raw text + metadata
   (page_label, etc.)
  - Failure: raises IngestionError, pipeline sets status=failed and returns

  Step 2: Chunk (ingestion/chunk.py)

  - SentenceSplitter(chunk_size=512, chunk_overlap=64, tokenizer=cl100k_base) вҖ”
  LlamaIndex splits the docs
  - For every chunk: assigns a deterministic ID "{document_id}:{index:04d}"
  (e.g. "abc123:0007")
  - Sets metadata on each node: {document_id, filename, chunk_index, 
  source_page}
  - This metadata is what gets stored in Chroma and later surfaces in Hit
  objects
  - Failure: raises IngestionError

  Step 3: Embed (ingestion/embed.py)

  - Takes the list of TextNodes, batches them in groups of 100
  - Calls OpenAI SDK directly:
  client.embeddings.create(model="text-embedding-3-small", input=[texts])
  - Retries up to 3Г— with exponential backoff (1s, 2s, 4s) on failure
  - Returns a parallel list of list[float] embedding vectors
  - Failure: raises IngestionError after all retries

  Step 4: Index (ingestion/index.py)

  - Creates/gets the rag_stage1 collection in chromadb.PersistentClient
  - collection.upsert(ids=..., embeddings=..., documents=[chunk texts], 
  metadatas=[...]) вҖ” stores everything
  - Collection uses hnsw:space: cosine so distances are in [0, 2]
  - Failure: raises IngestionError

  At the end: SQLite Document.status = ready, num_chunks = N.

  ---
  Stage 3 вҖ” SSE Progress Stream (api/documents.py)

  Client вҶ’ GET /api/documents/{id}/progress   (keeps connection open)

  The SSE generator polls SQLite every 0.3s:
  - Reads new IngestionEvent rows it hasn't sent yet вҶ’ streams them as event: 
  step SSE events
  - When Document.status == ready вҶ’ sends event: done and closes
  - When Document.status == failed вҶ’ sends event: error and closes

  Client sees a stream like:
  event: step    data: {"step":"upload","state":"complete","progress_pct":100}
  event: step    data: {"step":"parse","state":"running","progress_pct":10}
  event: step    data: {"step":"parse","state":"complete","progress_pct":30}
  event: step    data: {"step":"chunk","state":"running","progress_pct":30}
  ...
  event: done    data: {"document_id":"...","num_chunks":42,"num_pages":7}

  ---
  Stage 4 вҖ” Retrieval Pipeline (retrieval/)

  This is what was built this session. Entry point is retrieve(query):

  Step 4a: Rewrite (retrieval/rewrite.py)

  - Loads prompts/rewrite.txt once at module import вҖ” hard fails if missing
  - Calls OpenAI SDK directly:
  client.chat.completions.create(model="gpt-4o-mini", temperature=0.0, 
  max_tokens=200)
  - System message = the prompt file; user message = the raw question
  - Returns RewriteResult(original_query, rewritten_query, is_noop, 
  rewrite_fallback)
  - Fallback: on any exception вҶ’ returns original query, sets
  rewrite_fallback=True, logs WARNING. Never raises.

  Step 4b: Semantic Search (retrieval/semantic.py)

  - Embed: calls OpenAI SDK directly:
  client.embeddings.create(model="text-embedding-3-small", input=[query]) вҶ’
  1536-dim vector
  - Query: calls collection.query(query_embeddings=[embedding], n_results=10, 
  include=["documents","metadatas","distances"]) directly on the chromadb client
  - Score conversion: ChromaDB cosine distance вҲҲ [0, 2] вҶ’ similarity вҲҲ [0, 1]:
  score = 1 - distance / 2
  - Maps each result row to a Hit object using the metadata stored during
  chunking (document_id, filename, chunk_index, source_page)
  - Failure: raises RetrievalError вҖ” caller handles it

  Step 4c: Orchestrate (retrieval/retrieve.py)

  retrieve(query)
    в”Ӯ
    в”ңв”Җ rewrite(query) вҶ’ RewriteResult          # never raises
    в”Ӯ
    в””в”Җ semantic_search(rewritten_query, top_k=10)
         в”ңв”Җ success вҶ’ RetrievalResult(hits=[...], flags={})
         в””в”Җ exception вҶ’ RetrievalResult(hits=[], flags={"semantic_fallback":
  True})

  Merges rewrite_fallback and semantic_fallback into one flags dict on the
  returned RetrievalResult.
