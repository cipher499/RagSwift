  ---
  1. Which files call fetch directly and which do not

  - lib/api.ts ‚ÄĒ calls raw fetch (all non-SSE HTTP requests)
  - lib/sse.ts ‚ÄĒ calls fetchEventSource (SSE streams)
  - All components ‚ÄĒ never call fetch or fetchEventSource directly; they go
  through api.* and streamMessage/subscribeIngestionProgress

  ---
  2. How trace_partial events are disambiguated

  Inside lib/sse.ts, when ev.event === "trace_partial", inspect the parsed data
  object:
  - "rewritten_query" in data ‚Üí call onRewrite
  - "semantic_hits" in data ‚Üí call onHits

  Key check, not type assertion. Two separate trace_partial events arrive in
  that fixed order per specs/api.md ¬ß4.5.

  ---
  3. When the trace panel populates vs when done fires

  The trace panel populates progressively, before done:
  - Section 1 (Rewrite): fills on the first trace_partial (rewritten_query) ‚ÄĒ
  before any tokens
  - Section 2 (Hits): fills on the second trace_partial (semantic_hits) ‚ÄĒ before
   any tokens
  - Section 3 (Flags) + Section 4 (Meta / latency / langsmith URL): fill after 
  done, because they come from api.getTrace() which is called only after done
  fires, and the full Trace object is what carries flags, latency_ms, and
  langsmith_run_url

  ---
  4. What finalizeStream does

  Per specs/frontend.md ¬ß5:

  1. Appends the completed assistant Message to messages[chatId] (replacing the
  streaming bubble with a persisted entry)
  2. Stores the full Trace in traces[trace.id]
  3. Calls clearStream(), which resets streamingChatId ‚Üí null, partialTokens ‚Üí 
  "", partialTrace ‚Üí {}

  It is called from chat-area.tsx after api.getTrace() resolves following the
  done event.
